# Стенограф — Промпты извлечения артефактов

> Промпты для Claude API (Sonnet 4.6). Каждый получает стенограмму + метаданные встречи.
> Все промпты возвращают JSON для программной обработки.

---

## Общий контекст (system prompt для всех)

```
Ты — опытный аналитик IT-проектов, специализирующийся на внедрении корпоративных систем (1С, ERP, CRM).
Ты анализируешь стенограммы рабочих встреч и извлекаешь структурированную информацию.

Правила:
- Извлекай ТОЛЬКО то, что явно сказано на встрече. Не додумывай.
- Если информация неоднозначна — отмечай как "требует уточнения".
- Сохраняй терминологию заказчика (как они называют вещи).
- Для каждого извлечённого элемента указывай таймкод источника.
- Ответ строго в JSON-формате, без markdown-обёрток.
```

## Промпт 1: Протокол встречи

### Вход
```json
{
  "meeting_type": "обследование",
  "project_name": "Альфа-Логистик",
  "meeting_date": "2026-03-14",
  "transcript": "<полная стенограмма с таймкодами>"
}
```

### Промпт
```
Проанализируй стенограмму рабочей встречи и составь формальный протокол.

Тип встречи: {{meeting_type}}
Проект: {{project_name}}
Дата: {{meeting_date}}

Стенограмма:
{{transcript}}

Извлеки и верни JSON:

{
  "title": "Автоматически сгенерированное название встречи (краткое, по сути)",
  "meeting_type": "{{meeting_type}}",
  "date": "{{meeting_date}}",
  "duration_minutes": <длительность в минутах>,

  "participants": [
    {
      "name": "ФИО или как представился",
      "role": "должность / роль (если упоминалась)",
      "organization": "компания (если упоминалась)",
      "first_mention_at": "MM:SS"
    }
  ],

  "agenda": [
    {
      "topic": "Тема обсуждения",
      "discussed_from": "MM:SS",
      "discussed_to": "MM:SS"
    }
  ],

  "decisions": [
    {
      "id": "D-001",
      "description": "Что решили",
      "rationale": "Почему так решили (если обсуждалось)",
      "responsible": "Кто отвечает",
      "deadline": "Срок (если назван, иначе null)",
      "timestamp": "MM:SS",
      "confidence": "high|medium|low"
    }
  ],

  "action_items": [
    {
      "id": "AI-001",
      "description": "Что нужно сделать",
      "responsible": "Кто делает",
      "deadline": "Срок (если назван)",
      "priority": "high|medium|low",
      "timestamp": "MM:SS"
    }
  ],

  "key_statements": [
    {
      "speaker": "Кто сказал",
      "quote": "Дословная или близкая к дословной цитата",
      "context": "К чему относится",
      "timestamp": "MM:SS",
      "importance": "high|medium"
    }
  ],

  "next_steps": {
    "next_meeting_date": "Дата (если обсуждалась)",
    "next_meeting_topics": ["Что обсудить в следующий раз"],
    "homework": ["Что подготовить к следующей встрече"]
  }
}

Правила:
- decisions.confidence = "high" если решение было чётко подтверждено, "low" если было сказано предварительно.
- Если участник не представился по имени, используй "Участник 1", "Участник 2" и т.д.
- key_statements — только действительно важные высказывания, не более 10.
- Если deadline не назван явно, пиши null, не придумывай.
```

## Промпт 2: Требования (ТЗ)

### Промпт
```
Проанализируй стенограмму встречи и извлеки функциональные и нефункциональные требования.

Тип встречи: {{meeting_type}}
Проект: {{project_name}}

Стенограмма:
{{transcript}}

{{#if previous_requirements}}
Требования из предыдущих встреч (для сопоставления):
{{previous_requirements}}
{{/if}}

Извлеки и верни JSON:

{
  "functional_requirements": [
    {
      "id": "FR-001",
      "title": "Краткое название требования",
      "description": "Детальное описание: что система должна делать",
      "user_story": "Как [роль], я хочу [действие], чтобы [результат]",
      "acceptance_criteria": [
        "Критерий приёмки 1",
        "Критерий приёмки 2"
      ],
      "priority": "must|should|could|wont",
      "source_quote": "Цитата из стенограммы, откуда взято",
      "timestamp": "MM:SS",
      "status": "new|changed|confirmed|contradicts",
      "contradicts_id": "FR-XXX (если противоречит предыдущему)",
      "notes": "Дополнительные заметки"
    }
  ],

  "non_functional_requirements": [
    {
      "id": "NFR-001",
      "category": "performance|security|usability|reliability|scalability|integration",
      "title": "Краткое название",
      "description": "Описание: какой уровень качества/ограничение",
      "measurable_criteria": "Количественный критерий (если есть)",
      "timestamp": "MM:SS",
      "priority": "must|should|could"
    }
  ],

  "business_rules": [
    {
      "id": "BR-001",
      "rule": "Описание бизнес-правила",
      "condition": "При каком условии срабатывает",
      "action": "Что должно произойти",
      "exceptions": ["Исключения из правила"],
      "timestamp": "MM:SS"
    }
  ],

  "integrations": [
    {
      "id": "INT-001",
      "system": "Название внешней системы",
      "direction": "inbound|outbound|bidirectional",
      "data": "Какие данные передаются",
      "frequency": "Как часто (если обсуждалось)",
      "timestamp": "MM:SS"
    }
  ],

  "constraints": [
    {
      "type": "technical|business|regulatory|timeline",
      "description": "Описание ограничения",
      "timestamp": "MM:SS"
    }
  ],

  "process_description": {
    "as_is": "Описание текущего процесса (как работают сейчас)",
    "pain_points": ["Проблема 1", "Проблема 2"],
    "to_be_hints": "Намёки на желаемый процесс (если обсуждалось)"
  }
}

Правила:
- Приоритеты по MoSCoW: must = обязательно, should = желательно, could = было бы неплохо, wont = отложено.
- Если есть previous_requirements, сравни: status = "confirmed" (повторили), "changed" (изменили), "contradicts" (противоречит).
- Для новых требований (без предыдущих) status = "new".
- user_story не для всех FR — только где есть явный пользовательский контекст.
- business_rules — явные правила типа "если сумма > 100к, нужно согласование директора".
```

## Промпт 3: Карта рисков

### Промпт
```
Проанализируй стенограмму встречи и выяви проектные риски, неопределённости и противоречия.

Проект: {{project_name}}

Стенограмма:
{{transcript}}

{{#if previous_risks}}
Ранее выявленные риски:
{{previous_risks}}
{{/if}}

Извлеки и верни JSON:

{
  "risks": [
    {
      "id": "RISK-001",
      "category": "scope|technical|organizational|timeline|budget|integration|data",
      "title": "Краткое название риска",
      "description": "Детальное описание: что может пойти не так",
      "trigger": "Что спровоцирует этот риск",
      "impact": "high|medium|low",
      "probability": "high|medium|low",
      "mitigation_hint": "Намёк на решение (если обсуждалось на встрече)",
      "source_quote": "Цитата из стенограммы",
      "timestamp": "MM:SS",
      "status": "new|persists|resolved|escalated"
    }
  ],

  "uncertainties": [
    {
      "id": "UNC-001",
      "topic": "О чём неопределённость",
      "what_is_unknown": "Что именно непонятно",
      "who_can_clarify": "Кто может прояснить (если упоминалось)",
      "impact_if_unresolved": "Что будет, если не прояснить",
      "timestamp": "MM:SS"
    }
  ],

  "contradictions": [
    {
      "id": "CONTR-001",
      "statement_a": {
        "speaker": "Кто сказал",
        "position": "Что сказал",
        "timestamp": "MM:SS"
      },
      "statement_b": {
        "speaker": "Кто сказал другое",
        "position": "Что сказал",
        "timestamp": "MM:SS"
      },
      "resolution": "Как разрешили (null если не разрешили)",
      "severity": "high|medium|low"
    }
  ],

  "assumptions": [
    {
      "id": "ASM-001",
      "assumption": "Что предполагается без подтверждения",
      "stated_by": "Кто предположил",
      "needs_validation": true,
      "timestamp": "MM:SS"
    }
  ]
}

Правила:
- Ищи НЕЯВНЫЕ риски: "ну, это мы потом решим" → риск откладывания.
- Ищи противоречия: разные участники говорят разное о том же.
- Ищи допущения: "я думаю, что..." без подтверждения.
- impact × probability = приоритет, но вычислять не нужно.
- Если risk из previous_risks упоминался снова — status = "persists".
```

## Промпт 4: Глоссарий терминов

### Промпт
```
Проанализируй стенограмму встречи и извлеки термины предметной области.

Проект: {{project_name}}

Стенограмма:
{{transcript}}

{{#if previous_glossary}}
Существующий глоссарий проекта:
{{previous_glossary}}
{{/if}}

Извлеки и верни JSON:

{
  "terms": [
    {
      "id": "TERM-001",
      "term": "Термин как его используют",
      "aliases": ["Синонимы, если были"],
      "definition": "Определение на основе контекста",
      "domain": "logistics|finance|hr|production|sales|it|other",
      "usage_example": "Пример использования из стенограммы",
      "related_terms": ["Связанные термины"],
      "first_mention": "MM:SS",
      "mentioned_by": "Кто использовал",
      "confidence": "high|medium|low",
      "status": "new|updated|confirmed"
    }
  ],

  "abbreviations": [
    {
      "abbreviation": "ТМЦ",
      "full_form": "Товарно-материальные ценности",
      "context": "Где использовалось",
      "timestamp": "MM:SS"
    }
  ],

  "entity_mapping": [
    {
      "business_name": "Как называет заказчик",
      "system_name": "Как это в системе (1С, ERP — если обсуждалось)",
      "notes": "Расхождения в терминологии"
    }
  ]
}

Правила:
- Извлекай СПЕЦИФИЧНЫЕ термины предметной области, не общие слова.
- entity_mapping — критично для 1С-проектов: заказчик говорит "накладная", в 1С это "Реализация товаров".
- confidence = "low" если термин использовался один раз без объяснения.
- Если термин из previous_glossary повторился — status = "confirmed", если изменился — "updated".
- aliases — когда разные участники называют одно и то же по-разному.
```

### Промпт
```
Отформатируй сырую стенограмму в читаемый документ.

Сырая стенограмма с таймкодами:
{{raw_transcript}}

Информация о спикерах (если есть):
{{speakers}}

Обработай и верни JSON:

{
  "formatted_transcript": [
    {
      "timestamp": "00:00:15",
      "speaker": "Иванов И.И.",
      "text": "Отформатированный текст с пунктуацией и абзацами",
      "topics": ["тема1", "тема2"]
    }
  ],

  "chapters": [
    {
      "title": "Название раздела встречи",
      "start": "00:00:00",
      "end": "00:15:30",
      "summary": "Краткое содержание этого раздела (1-2 предложения)"
    }
  ],

  "statistics": {
    "total_duration_minutes": 97,
    "speakers_count": 4,
    "speaker_time": {
      "Иванов И.И.": 45,
      "Петрова А.С.": 30,
      "Сидоров К.В.": 15,
      "Неизвестный": 7

    "topics_discussed": ["тема1", "тема2", "тема3"],

Правила:
- Разбей на смысловые chapters (главы) по темам обсуждения.
- Исправь очевидные ошибки распознавания (артикуляция, имена собственные).
- НЕ меняй смысл — только форматирование и пунктуацию.
- speaker_time — в процентах от общего времени.
- Если спикеры не определены — используй "Спикер 1", "Спикер 2".
```

## Конфигурация вызовов

### Параметры Claude API для каждого промпта

```json
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 8192,
  "temperature": 0.1,
  "system": "<общий system prompt>",
  "messages": [
    {"role": "user", "content": "<промпт с подставленными данными>"}
  ]
}
```

- **temperature: 0.1** — минимальная креативность, максимальная точность
- **max_tokens: 8192** — достаточно для самого большого артефакта
- Промпты 1-5 запускаются **параллельно** (не зависят друг от друга)
- Промпт 6 (стенограмма) может запуститься **первым** (не зависит от извлечения)

### Обработка длинных стенограмм (> 100K токенов)

Для встреч длиннее 2 часов стенограмма может превышать контекстное окно.

**Стратегия:**
1. Разбить стенограмму на секции по 60K токенов с перекрытием 5K
2. Запустить промпт для каждой секции
3. Объединить результаты, дедуплицировать по id
4. Для требований: пересечения → подтверждение (confidence up)
5. Для глоссария: объединить, убрать дубли

### Кумулятивная логика (серия встреч)

При обработке N-й встречи в проекте:

1. Загрузить предыдущие артефакты (requirements, glossary, risks, questions)
2. Передать как `previous_*` в промпты
3. Промпт сам определит статусы: new / confirmed / changed / contradicts
4. После получения → merge в кумулятивные документы:
   - `cumulative/requirements.json` += новые FR, обновить changed
   - `cumulative/glossary.json` += новые термины
   - `cumulative/risks.json` += новые, обновить persists/resolved
   - Открытые вопросы — убрать отвеченные

## Промпты для типов встреч (адаптация)

### Модификаторы по типу встречи

Добавляются к system prompt:

**Предпроектное обследование:**
```
Фокус: процессы AS-IS, боли, требования, терминология.
Повышенное внимание: глоссарий, business_rules, process_description.
Типичные паттерны: "сейчас мы делаем так...", "проблема в том, что...", "хотелось бы..."
```

**Рабочая встреча:**
```
Фокус: задачи, статусы, блокеры, решения.
Повышенное внимание: action_items, decisions, deadlines.
Типичные паттерны: "нужно сделать...", "готово на X%", "заблокировано из-за..."
```

**Демонстрация:**
```
Фокус: замечания, доработки, приёмка.
Повышенное внимание: open_questions (замечания), decisions (принято/отклонено).
Типичные паттерны: "это не так работает", "добавьте ещё...", "это ОК, принято"
```

**Приёмка этапа:**
```
Фокус: формальная приёмка, перечень доработок, подтверждение.
Повышенное внимание: decisions (принято/на доработку), action_items (доработки).
Типичные паттерны: "этап принят с замечаниями", "доработать до...", "подтверждаю"
```
