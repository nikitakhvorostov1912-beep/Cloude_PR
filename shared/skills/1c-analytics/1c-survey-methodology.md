# Методология предпроектного обследования 1С

Ты — эксперт по предпроектному обследованию предприятий для внедрения 1С. Используй эту базу знаний при анализе процессов, составлении документации и GAP-анализе.

## Нормативная база

### ГОСТ 34.601-90 — Стадии создания АС
Стадия "Обследование объекта" включает:
1. Сбор данных об объекте автоматизации
2. Оценка качества функционирования объекта
3. Выявление проблем, решение которых возможно средствами автоматизации
4. Оценка целесообразности создания АС

**Артефакты стадии обследования:**
- Отчёт об обследовании (ТО)
- Тактико-техническое задание (ТТЗ) или Техническое задание (ТЗ)

### ГОСТ 34.602-89 — Требования к ТЗ на АС
Структура ТЗ:
1. Общие сведения
2. Назначение и цели создания системы
3. Характеристика объекта автоматизации
4. Требования к системе
5. Состав и содержание работ по созданию системы
6. Порядок контроля и приёмки
7. Требования к составу и содержанию работ по подготовке объекта автоматизации к вводу системы в действие
8. Требования к документированию
9. Источники разработки

## 1С:Методология проектного управления

### Этапы проекта 1С
1. **Инициация** — определение целей, границ, команды
2. **Обследование** — AS-IS процессы, требования
3. **Проектирование** — TO-BE процессы, ФТ, ТЗ
4. **Разработка** — настройка, доработка конфигурации
5. **Тестирование** — функциональное, интеграционное, пользовательское
6. **Внедрение** — миграция данных, обучение, опытная эксплуатация
7. **Сопровождение** — поддержка, доработки

### Артефакты обследования по 1С:ПМ
- **Карточка процесса** — стандартный формат описания:
  | Поле | Описание |
  |------|----------|
  | Название | Полное название процесса |
  | Код | Уникальный идентификатор |
  | Владелец | Ответственный за процесс |
  | Триггер | Событие запуска |
  | Входы | Документы, данные на входе |
  | Выходы | Результат процесса |
  | Участники | Роли и подразделения |
  | Шаги | Последовательность действий |
  | Решения | Точки ветвления |
  | Системы | Используемые ИТ-системы |
  | Проблемы | Текущие болевые точки |
  | Метрики | KPI процесса |

- **Лист требований** — классификация:
  - FR (Functional Requirements) — функциональные
  - NFR (Non-Functional Requirements) — нефункциональные
  - IR (Integration Requirements) — интеграционные
  - Приоритизация: MoSCoW (Must/Should/Could/Won't)

## Типовые процессы по отраслям

### Торговля (оптовая/розничная)
- Управление заказами клиентов (CRM → Заказ → Отгрузка → Оплата)
- Управление закупками (Заявка → Заказ поставщику → Приёмка → Оплата)
- Складской учёт (Приёмка → Размещение → Отбор → Отгрузка)
- Ценообразование (Прайс-листы → Скидки → Наценки)
- Взаиморасчёты (Дебиторка/Кредиторка → Сверка → Оплата)

### Производство
- Планирование производства (MRP/MRP II → Заказ на производство)
- Управление спецификациями (BOM → Технологические карты)
- Оперативное управление (Задания на производство → Выпуск)
- Управление качеством (Входной контроль → ОТК → Сертификация)
- Учёт затрат (Калькуляция себестоимости)
- ТОиР (Планирование ремонтов → Заявки → Выполнение)

### Строительство
- Управление проектами/объектами
- Сметный учёт (КС-2, КС-3)
- Управление субподрядчиками
- Учёт материалов на объектах

### Услуги
- Управление заявками (Help Desk / Service Desk)
- SLA и контроль сроков
- Выставление актов и счетов
- Управление проектами

## Антипаттерны обследования

### Что обычно упускают
1. **Исключения из процесса** — только "happy path", не спрашивают про нештатные ситуации
2. **Ручные обходные пути** — сотрудники делают в Excel/на бумаге то, что система не поддерживает
3. **Интеграции** — не выясняют все системы, с которыми нужна связь
4. **Объёмы данных** — количество документов/операций в день/месяц
5. **Отчётность** — какие отчёты нужны руководству, в какие сроки
6. **Права доступа** — кто что может видеть/редактировать
7. **Мобильный доступ** — нужен ли доступ с телефона/планшета
8. **Сезонность** — пиковые нагрузки
9. **Регуляторные требования** — ФСРАР, ВетИС, Меркурий, маркировка
10. **Миграция данных** — какие данные переносить из старых систем

### Типичные ошибки
- Записывать только что говорит руководитель (а надо и исполнителей)
- Не фиксировать pain points ("и так понятно")
- Не выяснять метрики процесса (сколько, как часто, как долго)
- Не спрашивать про будущие планы (рост, новые направления)
- Не документировать текущие системы и их роль

## Использование

При анализе транскриптов:
1. Определи отрасль и масштаб предприятия
2. Используй типовые процессы как чек-лист — все ли упомянуты?
3. Проверь антипаттерны — не упущено ли что-то?
4. Классифицируй требования по FR/NFR/IR и MoSCoW
5. Заполни карточки процессов по стандартному формату
