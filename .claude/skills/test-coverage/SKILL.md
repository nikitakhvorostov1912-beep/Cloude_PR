# /test-coverage — Анализ тестового покрытия

## Когда использовать
После написания новой функциональности, перед мержем.

## Инструкции

### 1. Инвентаризация тестов
- Найди все файлы `test_*.py` и `*.test.ts` / `*.test.tsx`
- Составь карту: какой модуль покрыт, какой нет

### 2. Backend (Python)
1. Запусти `python -m pytest tests/ -v --tb=short` из `projects/survey-automation/backend/`
2. Проверь:
   - Все API endpoints имеют тесты (GET, POST, PUT, DELETE)?
   - Edge cases покрыты (пустые данные, невалидный ввод, 404)?
   - Тесты изолированы (не зависят друг от друга)?
   - Нет sleep/time.sleep в тестах?
   - Fixtures переиспользуются через conftest.py?
3. Оцени покрытие критических путей:
   - Создание/удаление проектов
   - Загрузка файлов
   - Транскрипция
   - Анализ процессов
   - Генерация документов

### 3. Frontend (TypeScript)
1. Есть ли тесты вообще? Проверь наличие `jest.config`, `vitest.config`, `*.test.tsx`
2. Если есть — запусти и проверь результат
3. Критические компоненты должны быть покрыты:
   - Формы (валидация, submit)
   - API вызовы (success, error, loading states)
   - Навигация

### 4. Качество тестов
- Тесты проверяют ПОВЕДЕНИЕ, не реализацию?
- Есть negative tests (что НЕ должно работать)?
- Нет тестов, которые всегда pass (пустые assert)?

## Формат отчёта

```
## 🧪 Анализ тестового покрытия

### Backend
- Тестов: {N}
- Прошло: {N} ✅ / Упало: {N} ❌
- Покрытие endpoints: {X}%
- Непокрытые модули: {список}

### Frontend
- Тестов: {N} (или "Тесты отсутствуют ⚠️")

### Качество
- Изоляция: OK / Проблемы
- Edge cases: Покрыты / Не покрыты
- Negative tests: Есть / Нет

### Рекомендации
{что нужно добавить}

### Вердикт: PASS / FAIL
```
