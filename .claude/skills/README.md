
## 📝 Детальное описание каждого скилла

### `/full-review` — Полный ревью проекта
**Когда использовать:** Перед релизом, при приёмке проекта, для общей оценки состояния.

Последовательно запускает: health-check → architecture-review → find-bugs → ui-ux-review → lint-check. Результат — единый отчёт с оценками по 10-балльной шкале и планом действий на 3 недели.

---

### `/architecture-review` — Ревью архитектуры
**Когда использовать:** В начале работы над проектом, при рефакторинге, при добавлении крупной фичи.

Анализирует:
- Структуру проекта и модулей
- Граф зависимостей (ищет циклические)
- Поток данных от входа до выхода
- Контракты между модулями
- Конфигурацию и дефолты
- Масштабируемость (память, таймауты, большие данные)

Выдаёт оценку X/10 с разделением на критические проблемы, предупреждения и рекомендации.

---

### `/find-bugs` — Поиск багов
**Когда использовать:** Регулярно, перед релизом, после крупных изменений.

Ищет по 5 категориям:
1. **Runtime** — необработанные исключения, None-доступы, утечки ресурсов
2. **Логические** — ошибки условий, off-by-one, несовпадение дефолтов
3. **Интеграция** — таймауты API, ошибки авторизации, отсутствие файлов
4. **Безопасность** — path traversal, инъекции, секреты в коде
5. **Платформа** — пути Windows/Linux, кодировки, длинные пути

Каждый баг получает ID (BUG-001) и приоритет (Critical/High/Medium/Low).

---

### `/smart-fix` — Автоисправление багов
**Когда использовать:** После `/find-bugs`, когда нужно быстро исправить найденное.

Берёт результаты `/find-bugs` и автоматически исправляет:
- Сначала критические (крэши, потеря данных)
- Затем высокий приоритет
- Минимальные изменения, не ломающие существующий код
- Добавляет обработку ошибок и type hints

---

### `/ui-ux-review` — UI/UX аудит
**Когда использовать:** После создания UI, перед демонстрацией заказчику.

Проверяет по 10 эвристикам Нильсена:
- Видимость состояния (прогресс, загрузка)
- Терминология понятна пользователю
- Возможность отмены и возврата
- Консистентность паттернов
- Предотвращение ошибок
- Подсказки и помощь
- Доступность (контраст, размеры, Tab-навигация)

Выдаёт постраничный разбор + список быстрых улучшений.

---

### `/tdd` — Test-Driven Development
**Когда использовать:** При разработке новых модулей, при рефакторинге.

Создаёт полный набор тестов в `tests/`:
- conftest.py с общими фикстурами
- Unit-тесты для каждого модуля
- Моки для внешних зависимостей (API, ML-модели, файлы)
- Интеграционные тесты
- Запускает pytest и сообщает результаты

---

### `/e2e-test` — E2E тестирование
**Когда использовать:** Для автоматизации проверки веб-интерфейса.

Создаёт E2E тесты через Playwright:
- Автозапуск веб-сервера
- Тест каждой страницы
- Тест навигации и сохранения состояния
- Тест полного пользовательского пути
- Скриншоты при ошибках

---

### `/lint-check` — Линтинг и типы
**Когда использовать:** После каждого значительного изменения кода.

Запускает:
- **ruff** — стиль, неиспользуемые импорты, типичные ошибки, автоисправление
- **ruff format** — единообразное форматирование
- **pyright** — статическая проверка типов
- Создаёт конфиг ruff если отсутствует

---

### `/debug` — Систематическая отладка
**Когда использовать:** При конкретной ошибке. Передай описание проблемы как аргумент.

Процесс:
1. Воспроизведение ошибки
2. Изоляция — трассировка потока данных до точки сбоя
3. Анализ корневой причины (почему, не что)
4. Минимальное исправление
5. Верификация

---

### `/health-check` — Диагностика системы
**Когда использовать:** При первом запуске, после установки, когда что-то не работает.

Проверяет:
- Версию Python и установленные пакеты
- Доступность внешних сервисов (Ollama, API, БД)
- Системные утилиты (FFmpeg, Node.js и т.д.)
- Валидность конфигурации
- Структуру директорий и права
- Импорт всех модулей проекта

Выдаёт таблицу статусов: ЗДОРОВ / ЧАСТИЧНО / СЛОМАН.
