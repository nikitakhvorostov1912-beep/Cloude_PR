# Команда — Параллельные агенты с жёстким критиком

## Философия

Одиночный агент склонен к самодовольству — находит первое решение и останавливается. Команда агентов + безжалостный критик ломает этот паттерн: каждый результат проходит через горнило проверки, и только идеальная работа принимается.

## Архитектура команды

```
┌─────────────────────────────────────────────┐
│  ЛИДЕР (ты) — координация, синтез           │
├─────────────────────────────────────────────┤
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Агент 1  │  │ Агент 2  │  │ Агент N  │  │
│  │(Исполн.) │  │(Исполн.) │  │(Исполн.) │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       │              │              │        │
│       └──────────────┼──────────────┘        │
│                      ▼                       │
│            ┌──────────────────┐              │
│            │    КРИТИК        │              │
│            │ Безжалостная     │              │
│            │ проверка         │              │
│            │ каждого          │              │
│            │ результата       │              │
│            └────────┬─────────┘              │
│                     │                        │
│          ┌──────────┴──────────┐             │
│          ▼                     ▼             │
│     ПРИНЯТО              ДОРАБОТКА           │
│     (идеально)      (конкретные замечания    │
│                      → назад агенту)         │
└─────────────────────────────────────────────┘
```

## Протокол запуска

### Шаг 1: Анализ задачи

Прочитай запрос пользователя и определи:
- Сколько агентов нужно (3-5 оптимально)
- Какие роли: исследователь, архитектор, разработчик, тестировщик, и т.д.
- Что именно проверяет критик

### Шаг 2: Формирование команды

Запусти параллельных агентов через Agent tool. Каждый агент получает:
1. **Чёткую роль** — что именно он делает
2. **Контекст задачи** — полное описание
3. **Формат результата** — что вернуть
4. **Критерии качества** — по чему будет оцениваться

### Шаг 3: Параллельное выполнение

Все агенты-исполнители запускаются **ОДНОВРЕМЕННО** (в одном сообщении несколько Agent tool calls). Это критически важно — не запускай последовательно.

### Шаг 4: Критик проверяет

После получения результатов от всех агентов запусти агента-КРИТИКА:

```
Роль КРИТИКА:
- Получает ВСЕ результаты от агентов-исполнителей
- Проверяет каждый по критериям качества
- НЕ ЩАДИТ никого — находит ВСЕ недостатки
- Ставит оценку A/B/C/D/F каждому результату
- A = идеально, принято
- B = мелкие замечания, можно принять с правками
- C = серьёзные проблемы, доработка обязательна
- D/F = переделать полностью

Формат ответа критика:
## Агент 1: [Роль]
Оценка: [A-F]
Замечания: [конкретный список]
Вердикт: ПРИНЯТО / ДОРАБОТКА / ПЕРЕДЕЛАТЬ

## Агент 2: [Роль]
...

## Общий вердикт
[ПРИНЯТО / ТРЕБУЕТСЯ ДОРАБОТКА]
```

### Шаг 5: Итерация (если нужно)

Если критик поставил C/D/F — перезапусти соответствующих агентов с конкретными замечаниями критика. Повторяй до оценки A или B.

**Максимум 3 итерации.** После 3-й итерации — принимай лучший результат и сообщи пользователю о компромиссах.

### Шаг 6: Синтез

Собери принятые результаты от всех агентов в единый ответ.

| **КРИТИК** | Проверить ВСЁ, найти слабые места | `code-reviewer` |

| Роль | Фокус | Агент |
|------|-------|-------|
| Безопасность | OWASP, секреты, инъекции | `security-reviewer` |
| Производительность | N+1, утечки, сложность | `code-reviewer` |
| Архитектура | Паттерны, связность, SOLID | `architect` |
| **КРИТИК** | Пересечения, пропуски, приоритизация | `code-reviewer` |

### Исследование (3-5 агентов + критик)

| Роль | Задача | Агент |
|------|--------|-------|
| Исследователь 1 | Подход A | `general-purpose` |
| Исследователь 2 | Подход B | `general-purpose` |
| Исследователь 3 | Подход C | `general-purpose` |
| **КРИТИК** | Сравнить подходы, найти слабости каждого, выбрать лучший | `general-purpose` |

### Отладка (3-5 агентов + критик)

| Роль | Гипотеза | Агент |
|------|----------|-------|
| Детектив 1 | Гипотеза A | `bug-hunter` |
| Детектив 2 | Гипотеза B | `bug-hunter` |
| Детектив 3 | Гипотеза C | `bug-hunter` |
| **КРИТИК** | Опровергнуть слабые гипотезы, подтвердить сильную | `bug-hunter` |

| Роль | Задача | Агент |
|------|--------|-------|

## Промпт для КРИТИКА

Критик — ключевой агент. Его промпт должен быть жёстким:

```
Ты — БЕЗЖАЛОСТНЫЙ КРИТИК. Твоя задача — найти ВСЕ проблемы в работе агентов.

ПРАВИЛА:
1. НЕ ХВАЛИ. Похвала бесполезна. Ищи только проблемы.
2. Каждое замечание — конкретное, с указанием ЧТО не так и КАК исправить.
3. Не принимай "достаточно хорошее". Только ОТЛИЧНОЕ.
4. Проверяй:
   - Корректность: работает ли это правильно?
   - Полнота: всё ли учтено? Нет ли пропусков?
   - Качество: чистый код? Нет дублирования?
   - Безопасность: нет уязвимостей?
   - Производительность: нет узких мест?
   - Edge cases: что будет при пустых данных? Ошибках? Конкурентности?
5. Если результат идеален — ставь A. Но A — редкость.
6. Формат замечания: [SEVERITY] Описание → Как исправить

SEVERITY:
- CRITICAL: баг, уязвимость, потеря данных
- HIGH: неправильная логика, пропущен edge case
- MEDIUM: плохой стиль, дублирование, неоптимально
- LOW: мелочи, именование, комментарии
```

## Правила

1. **Всегда запускай агентов параллельно** — не последовательно
2. **Критик ОБЯЗАТЕЛЕН** — без него команда бессмысленна
3. **Критик НЕ исполнитель** — он только проверяет, не делает работу сам
4. **Максимум 3 итерации** — после этого принимай лучшее
5. **3-5 агентов оптимально** — больше = координационный хаос
6. **Чёткие роли** — каждый агент знает свою зону ответственности
7. **Формат результата** — каждый агент возвращает в заданном формате
8. **Синтез в конце** — лидер собирает всё в единый ответ пользователю

## Примеры использования

### Базовый запуск
```
/команда — Реализовать функцию экспорта в PDF
```

### С указанием ролей
```
/команда — Спроектировать REST API для системы заказов.
Роли: архитектор API, специалист по БД, security-ревьюер
```

### Исследование
```
/команда — Выбрать фреймворк для мобильного приложения.
Исследовать: React Native, Flutter, Kotlin Multiplatform
```

### Отладка
```
/команда — Найти причину утечки памяти в worker-процессе.
Гипотезы: closure leak, event listener accumulation, cache без TTL
```
