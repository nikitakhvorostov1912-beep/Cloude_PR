## Принципы работы

- **Адаптивность**: количество агентов и глубина анализа зависят от сложности задачи
- **Ранняя валидация**: ревью плана до реализации, а не после
- **Уточнение требований**: выявление неоднозначностей до проектирования через вопросы пользователю
- **Атомарные шаги**: этапы реализации с критериями приемки и проверками
- **Отслеживание прогресса**: после каждой фазы отмечай её как завершённую

> Правила кода: `.claude/rules/1c/1c-rules.md`

## Phase 0: Инициализация и оценка сложности

**Цель**: понять масштаб задачи

**Действия**:
1. Создай список задач со всеми фазами
2. Создай директорию `.tasks/task-[feature-name]/`
3. **Оцени сложность** (простая/средняя/сложная/критичная):
   - **Простая**: небольшое изменение, очевидная реализация
   - **Средняя**: несколько модулей, требует понимания архитектуры
   - **Сложная**: несколько подсистем, неочевидные решения
   - **Критичная**: архитектурные изменения, влияние на всю систему
4. Запиши в `.tasks/task-[feature-name]/phase0-complexity.md`

## Phase 1: Discovery

**Цель**: понять, что нужно построить

**Действия**:
1. Если доработка неясна, спроси: какую проблему решают? что должна делать? ограничения?
2. Резюмируй понимание и получи подтверждение
3. Сохрани в `.tasks/task-[feature-name]/phase1-requirements.md`

---

## Phase 2: Исследование кодовой базы 1C

**Цель**: понять существующий код и паттерны

**Адаптивное исследование** (на основе сложности из Phase 0):

1. **Сколько агентов `1c-code-explorer` запустить?** (1-4+)
2. **Параллельно или последовательно?**
   - Параллельно — разные независимые области
   - Последовательно — углубление после находок
3. **Фокус**: похожие доработки, архитектура, интеграции, критичные компоненты
4. **Итеративное углубление**: запустить → посмотреть → решить нужно ли ещё

**Действия**:
1. Прими решение о стратегии исследования
2. Запусти агентов `1c-code-explorer` с путём к `phase1-requirements.md`
3. Прочитай все указанные ключевые файлы
4. Сохрани в `.tasks/task-[feature-name]/phase2-exploration.md`

---

## Phase 3: Уточняющие вопросы

**Цель**: заполнить пробелы ДО проектирования

**Действия**:
1. Просмотри находки и исходный запрос
2. Выяви: граничные случаи, обработка ошибок, интеграции, обратная совместимость, производительность
3. **Представь вопросы пользователю** в организованном списке
4. **Жди ответов**
5. Сохрани в `.tasks/task-[feature-name]/phase3-clarifications.md`

---

## Phase 4: Проектирование архитектуры

**Цель**: спроектировать архитектуру реализации

**Адаптивный подход** по сложности:

| Сложность | Агентов `1c-code-architect` | Подход |
|-----------|----------------------------|--------|
| Простая | 1 | Один практичный план |
| Средняя | 1 (опционально 2) | Мультисэмплинг если решение неочевидно |
| Сложная/Критичная | 2-3 | Параллельно: минимальные изменения / чистая архитектура / баланс |

**Действия**:
1. Запусти агентов с путями ко всем артефактам (phase1-3)
2. При нескольких подходах — представь пользователю с рекомендацией
3. Создай `.tasks/task-[feature-name]/phase4-architecture.md`:
   - Постановка задачи, оценка сложности
   - Выбранный подход с обоснованием
   - **Этапы реализации** (чеклист):
     - Простая = 1, средняя = 2-4, сложная = 4-8, критичная = 5-10+
     - Формат: `- [ ] **Этап N**: описание + файлы + критерии приемки`
   - Mermaid-диаграммы

---

## Phase 5: Ревью плана

**Цель**: валидировать план ДО реализации

**Действия**:
1. Запусти `1c-code-architect` для ревью:
   - Полнота, корректность, реалистичность, соответствие требованиям, зависимости
2. При проблемах — обнови план и повтори ревью
3. Сохрани в `.tasks/task-[feature-name]/phase5-plan-review.md`
4. **Представь план пользователю. Спроси: "Можем начинать?"**

**НЕ ПЕРЕХОДИ К PHASE 6 БЕЗ ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ**

---

## Phase 6: Реализация по этапам

**Действия** для каждого этапа:
1. Отметь `- [ ]` → `- [🔄]`
2. Запусти `1c-code-writer` с путём к плану и указанием этапа
3. **Для XML-артефактов** используй скиллы генерации:
   - Объекты метаданных → `/meta-compile` → `/meta-validate`
   - Управляемые формы → `/form-patterns` (справка) → `/form-compile` → `/form-validate`
   - СКД → `/skd-compile` → `/skd-validate`
   - Макеты → `/mxl-compile` → `/mxl-validate`
   - Роли → `/role-compile` → `/role-validate`
   - Подсистемы → `/subsystem-compile` → `/subsystem-validate`
   - Обработки → `/epf-init` → `/epf-add-form` → `/epf-build`
   - Отчёты → `/erf-init` → `/erf-build`
   - Расширения → `/cfe-init` → `/cfe-borrow` → `/cfe-patch-method`
4. После каждого compile/edit → обязательно validate
5. Проверь критерии приемки
6. Если не выполнены — запусти снова для исправления
7. Отметь `- [🔄]` → `- [x]`

> Полные цепочки вызовов: `.claude/rules/1c/1c-skill-orchestration.md`

---

## Phase 7: Ревью кода

**Действия** по размеру задачи:
- **Простая**: только `1c-code-reviewer`
- **Средняя/сложная**: `1c-code-architect` (соответствие плану) + `1c-code-reviewer` (качество)
- **Очень большая**: разбей по модулям, каждому — пару агентов

1. Запусти автоматическую валидацию XML: `/meta-validate`, `/form-validate`, `/skd-validate`, `/epf-validate` и др.
2. Сохрани в `.tasks/task-[feature-name]/phase7-code-review.md`
3. **Спроси пользователя**: исправить сейчас / позже / продолжить как есть
4. При "исправить сейчас" — `1c-code-writer` → повторное ревью
5. При наличии веб-клиента → `/web-test` для автоматического тестирования

---

## Phase 8: Итоги

**Действия**:
1. Отметь все задачи как завершённые
2. Сохрани `.tasks/task-[feature-name]/phase8-summary.md`:
   - Что построено (ссылки на этапы)
   - Ключевые решения
   - Изменённые файлы (`git diff --stat`)
   - Оставшийся технический долг
   - Следующие шаги
3. Представь резюме пользователю
