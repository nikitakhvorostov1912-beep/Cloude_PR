## Обзор

Помогает превращать идеи в полноценно проработанные дизайны и спецификации через совместный диалог.

Начните с понимания текущего контекста проекта, затем задавайте вопросы по одному. Когда поймёте, что именно нужно построить, представьте дизайн небольшими разделами (200-300 слов), уточняя на каждом этапе.

## Когда использовать

- Перед созданием фич, компонентов, модулей
- Когда идея сырая и нужно прояснить требования
- Перед `1c-feature-dev` для уточнения задачи
- Когда нужно сравнить несколько подходов

## Когда НЕ использовать

- Задача уже чётко определена → сразу `1c-feature-dev` или `planner`
- Нужен только code review → используй `code-reviewer`
- Техническое исследование → используй `deep-researcher`

## Процесс

**Понимание идеи:**
- Изучите текущее состояние проекта (файлы, документацию, коммиты)
- Задавайте вопросы по одному
- Предпочитайте вопросы с вариантами ответов
- В одном сообщении — только один вопрос
- Фокус: цели, ограничения, критерии успеха

**Исследование подходов:**
- Предложите 2-3 подхода с плюсами и минусами
- Начинайте с рекомендуемого варианта и объясняйте почему

**Представление дизайна:**
- Разбейте на разделы по 200-300 слов
- После каждого раздела спрашивайте, всё ли правильно
- Темы: архитектура, компоненты, поток данных, обработка ошибок
- Будьте готовы вернуться и уточнить

## После дизайна

Запишите утверждённый дизайн в `docs/plans/YYYY-MM-DD-<topic>-design.md`

## Ключевые принципы

- **Один вопрос за раз** — не перегружайте
- **Предпочтение вариантам** — проще отвечать
- **YAGNI безжалостно** — убирайте ненужные фичи
- **2-3 альтернативы** — всегда перед выбором
- **Инкрементальная валидация** — проверяйте каждый шаг
- **Гибкость** — возвращайтесь если что-то не так
