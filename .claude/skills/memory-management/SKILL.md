# Memory Management — Проактивная система памяти

## КОГДА ИСПОЛЬЗОВАТЬ

- `/memory save [что запомнить]` — сохранить факт/решение/предпочтение
- `/memory recall [тема]` — вспомнить всё по теме
- `/memory status` — обзор всей памяти
- `/memory predict` — предсказать что может понадобиться дальше
- Автоматически: при обнаружении важных решений, предпочтений, паттернов

## АРХИТЕКТУРА ПАМЯТИ (3 уровня)

Вдохновлено memU — файловая метафора:

```
memory/
├── categories/          # Уровень 3: Авто-категории (как папки)
│   ├── tech-stack.md    # Технологический стек проекта
│   ├── preferences.md   # Предпочтения пользователя
│   ├── decisions.md     # Принятые архитектурные решения
│   ├── patterns.md      # Выявленные паттерны работы
│   └── contacts.md      # Контакты, аккаунты, ключи
│
├── items/               # Уровень 2: Конкретные факты (как файлы)
│   ├── 2026-03-06_stack-choice.md
│   ├── 2026-03-06_api-key-locations.md
│   └── ...
│
└── resources/           # Уровень 1: Исходные данные (как mount points)
    ├── conversations/   # Ключевые моменты из разговоров
    ├── documents/       # Ссылки на важные документы
    └── links/           # Полезные URL с аннотациями
```

## ПРОТОКОЛ СОХРАНЕНИЯ

При обнаружении важной информации:

### 1. Классификация факта
Определи тип:
- **PREFERENCE** — предпочтение пользователя (язык, стиль, инструменты)
- **DECISION** — архитектурное/техническое решение с обоснованием
- **PATTERN** — повторяющийся паттерн поведения/работы
- **CREDENTIAL** — расположение ключей/токенов (НЕ сами ключи!)
- **CONTEXT** — контекст проекта (стек, структура, зависимости)
- **CONTACT** — контактные данные, аккаунты
- **LESSON** — урок из ошибки или успешного решения

### 2. Формат записи (Item)
```markdown
# [Краткий заголовок]

- **Тип:** DECISION | PREFERENCE | PATTERN | ...
- **Дата:** YYYY-MM-DD
- **Категории:** [tech-stack, preferences, ...]
- **Связи:** [ссылки на связанные items]
- **Источник:** [откуда получена информация]

## Факт
[Что именно запомнить — конкретно и однозначно]

## Контекст
[Почему это важно, при каких обстоятельствах возникло]

## Применение
[Когда и как использовать эту информацию]
```

### 3. Кросс-ссылки
- Каждый новый факт проверяется на связи с существующими
- Связанные факты ссылаются друг на друга
- Категории автоматически обновляются

## ПРОТОКОЛ ИЗВЛЕЧЕНИЯ (Recall)

При запросе `/memory recall [тема]`:

1. **Поиск по категориям** — найти релевантные категории
2. **Grep по items** — найти все факты по ключевым словам
3. **Кросс-ссылки** — подтянуть связанные факты
4. **Ранжирование** — от самых релевантных к менее
5. **Сводка** — компактный ответ с источниками

## ПРОАКТИВНОЕ ПОВЕДЕНИЕ (из memU)

### Автоматическое запоминание
Claude ДОЛЖЕН автоматически сохранять в память:
- Явные предпочтения пользователя ("мне нравится...", "используй...")
- Принятые решения ("давай сделаем через X")
- Найденные ключи/токены (ТОЛЬКО расположение, не значения!)
- Повторяющиеся запросы (паттерны)
- Ошибки и их решения (уроки)

### Предсказание потребностей
На основе накопленной памяти:
- Если пользователь начинает новый проект → подтянуть предпочтения стека
- Если возникла ошибка → проверить не было ли похожей в уроках
- Если спрашивает о технологии → проверить есть ли решение в decisions

## СЕССИОННОЕ ПЛАНИРОВАНИЕ (Per-Task Files)

Для сложных задач (3+ шагов) создавай файлы планирования в рабочей директории:

```
.claude/plans/
├── task_plan.md     # Фазы задачи, решения, ошибки
├── findings.md      # Находки из исследования
└── progress.md      # Лог прогресса по фазам
```

### task_plan.md — План задачи
```markdown
# Задача: [название]

## Цель
[Конкретный результат]

## Фазы
| # | Фаза | Статус | Результат |
|---|-------|--------|-----------|
| 1 | Исследование | ✅ complete | Найдено 3 подхода |
| 2 | Реализация | 🔄 in_progress | — |
| 3 | Тестирование | ⏳ pending | — |

## Решения
| Решение | Альтернативы | Причина выбора |
|---------|-------------|----------------|
| FastAPI | Flask, Django | Async + типизация |

## Ошибки
| # | Ошибка | Попыток | Решение |
|---|--------|---------|---------|
| 1 | Import error | 2 | Добавлен __init__.py |
```

### findings.md — Находки исследования
```markdown
# Находки: [тема]

## Требования
- [что должно быть]

## Технические решения
- [что нашли и выбрали]

## Ресурсы
- [полезные ссылки и документация]
```

### progress.md — Лог прогресса
```markdown
# Прогресс: [задача]

## Фаза 1: Исследование
- [действие 1] → [результат]
- [действие 2] → [результат]

## Результаты тестов
| Тест | Статус | Детали |
|------|--------|--------|
| pytest | ✅ PASS | 12/12 |
| build | ✅ PASS | 0 errors |

## Лог ошибок
| Время | Ошибка | Попытка # | Результат |
|-------|--------|-----------|-----------|
| 14:30 | ModuleNotFoundError | 1 | Установлен пакет |
| 14:35 | ModuleNotFoundError | 2 | Исправлен путь |
```

## ПРАВИЛО 2-ДЕЙСТВИЙ (2-Action Rule)

**После каждых 2 поисков/чтений — НЕМЕДЛЕННО сохрани находки в файл.**

Почему: информация, не записанная после 2 действий, рискует потеряться при compaction или переключении контекста.

Применение:
1. Прочитал 2 файла → записал ключевые факты в findings.md или memory
2. Сделал 2 WebSearch → зафиксировал результаты
3. Проанализировал 2 подхода → записал сравнение

**НЕ продолжай исследование, пока не зафиксировал текущие находки!**

## ЛОГИРОВАНИЕ ОШИБОК (Error Tracking)

При каждой ошибке в задаче фиксируй в progress.md или task_plan.md:

```markdown
| # | Ошибка | Попытка | Что пробовал | Результат |
|---|--------|---------|-------------|-----------|
| 1 | TypeError | 1 | Приведение типов | ❌ Не помогло |
| 1 | TypeError | 2 | Изменил схему | ✅ Решено |
```

Зачем:
- Не повторять провалившиеся подходы
- При compaction — лог сохранится в файле
- Ускоряет отладку при повторных сессиях

## ПРАВИЛА

1. **НИКОГДА не храни секреты** — только РАСПОЛОЖЕНИЕ ключей (файл, переменная)
2. **Датируй всё** — каждый факт с датой для актуальности
3. **Конкретность** — "использует Python 3.12" а не "использует Python"
4. **Кросс-ссылки** — связывай факты между собой
5. **Чистка** — устаревшие факты помечай как deprecated
6. **Русский язык** — вся память на русском
7. **Минимализм** — короткие точные записи, не эссе
8. **2-Action Rule** — после 2 действий поиска/чтения → сохрани находки
9. **Error Tracking** — логируй каждую ошибку с номером попытки
