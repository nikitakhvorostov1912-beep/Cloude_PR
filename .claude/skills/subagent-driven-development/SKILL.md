# Subagent-Driven Development

Выполнение плана через отдельные субагенты на каждую задачу с двухэтапной проверкой после каждой: соответствие спецификации, затем качество кода.

**Принцип:** Свежий субагент на задачу + двухэтапный review = высокое качество, быстрая итерация

## Когда использовать

- Есть готовый план реализации
- Задачи в основном независимы
- Работа в текущей сессии (не параллельная)

## Процесс

```
1. Прочитать план, извлечь ВСЕ задачи с полным текстом
2. Создать TodoWrite со всеми задачами

Per Task:
3. Запустить субагент-имплементатор (general-purpose)
   → Если вопросы — ответить, дать контекст
   → Имплементация, тесты, коммит, самопроверка
4. Запустить субагент-ревьюер спецификации
   → ✅ соответствует спеке → далее
   → ❌ проблемы → имплементатор исправляет → повторный review
5. Запустить субагент-ревьюер качества кода
   → ✅ одобрено → задача завершена
   → ❌ проблемы → имплементатор исправляет → повторный review
6. Отметить задачу как выполненную

After All Tasks:
7. Финальный code review всей реализации
```

## Промпт для имплементатора

```
Ты реализуешь Задачу N: [название]

## Описание задачи
[ПОЛНЫЙ ТЕКСТ задачи из плана]

## Контекст
[Где это вписывается, зависимости, архитектурный контекст]

## Перед началом
Если есть вопросы по требованиям, подходу, зависимостям — ЗАДАЙ ИХ СЕЙЧАС.

## Твоя работа
1. Реализуй ТОЧНО то что в задаче
2. Напиши тесты (TDD если указано)
3. Проверь что работает
4. Закоммить
5. Самопроверка (полнота, качество, YAGNI, тесты)
6. Отчёт: что сделал, тесты, файлы, проблемы
```

## Промпт для ревьюера спецификации

```
Проверь соответствует ли реализация спецификации.

## Что запрашивалось
[ПОЛНЫЙ ТЕКСТ требований]

## КРИТИЧЕСКИ: Не доверяй отчёту
Прочитай РЕАЛЬНЫЙ КОД и проверь:
- Пропущенные требования
- Лишняя/ненужная работа
- Неправильное понимание

Результат: ✅ соответствует | ❌ проблемы [список с file:line]
```

## Промпт для ревьюера качества

```
Проверь качество реализации (чистота, тесты, поддерживаемость).
Запускать ТОЛЬКО после прохождения spec review.

Результат: Сильные стороны, Проблемы (Critical/Important/Minor), Оценка
```

## Красные флаги — НИКОГДА

- Не начинать на main/master без согласия
- Не пропускать review (ни spec, ни quality)
- Не продолжать с неисправленными проблемами
- Не запускать параллельные субагенты-имплементаторы (конфликты)
- Не давать субагенту читать файл плана (передавай полный текст)
- Не начинать quality review ДО прохождения spec review
- Не переходить к следующей задаче пока есть открытые проблемы

## Преимущества

- Свежий контекст на каждую задачу (нет загрязнения)
- Двухэтапный review ловит проблемы рано
- Вопросы субагента всплывают ДО начала работы
- Быстрая итерация без human-in-loop между задачами
