# Requirements List — Генерация листа требований

Сформируй структурированный лист требований на основе бизнес-процессов и GAP-анализа.

## Классификация требований

### По типу
- **FR** (Functional Requirements) — функциональные: что система должна делать
- **NFR** (Non-Functional Requirements) — нефункциональные: производительность, безопасность, удобство
- **IR** (Integration Requirements) — интеграционные: связь с другими системами

### По приоритету (MoSCoW)
- **Must** — обязательно для запуска (без этого проект не имеет смысла)
- **Should** — важно, но можно отложить на 2-й этап
- **Could** — желательно, если хватит времени/бюджета
- **Won't** — не в этом проекте (зафиксировать для будущего)

### По категории
- Учёт и документооборот
- Отчётность и аналитика
- Интеграция
- Настройка и администрирование
- Миграция данных
- Обучение
- Печатные формы
- Права доступа

## Формат данных

```json
{
  "requirements": [
    {
      "id": "FR-001",
      "description": "Автоматическое создание заказа поставщику при снижении остатка ниже минимума",
      "type": "FR",
      "priority": "Must",
      "module": "Управление закупками",
      "category": "Учёт и документооборот",
      "source_process": "proc_003",
      "source_gap": "gap_005",
      "acceptance_criteria": "При остатке товара < min_stock создаётся документ 'Заказ поставщику' с расчётом количества",
      "effort_hours": 24,
      "status": "Новое"
    }
  ],
  "summary": {
    "total": 45,
    "by_type": {"FR": 30, "NFR": 10, "IR": 5},
    "by_priority": {"Must": 15, "Should": 18, "Could": 10, "Wont": 2},
    "total_effort_hours": 640
  }
}
```

## Правила

- Каждое требование привязано к процессу (source_process) и/или GAP (source_gap)
- Критерии приёмки обязательны для FR
- Трудозатраты в человеко-часах
- ID формат: {тип}-{номер} (FR-001, NFR-001, IR-001)
- Описание на русском, чёткое и однозначное
- Одно требование = одна единица функционала
